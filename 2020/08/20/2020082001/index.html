    也就是说synchronized关键词的确可以保证秒杀时该方法一直是一个线程在执行，但是它无法做到细粒度的控制，前面是将synchronized关键词加到了一个方法上，假设我们有很多商品，然后每个商品的id不同，但是它们都会访问之前那个下单的方法，且秒杀A商品的人数明细多余秒杀B商品的人数，这样一旦进入这个方法，它们两者都是很慢的。还有一个情况就是它只适用于单节点的情况。

    Redis缓存

    详细大家都听说过数据缓存层，尤其是在大流量的情况下，使用缓存可以有效的提高数据的读取速度。在缓存中一般会遇到这三个概念：命中、失效和更新。命中指的是应用程序从缓存中获取数据，取到后返回。因为缓存是有时间的，那么时间到了就会变成失效状态。应用程序把数据存到数据库中，再放到缓存中，这就是更新。

    举个例子来说，我们知道Java应用运行在Tomcat容器中，假如我们需要取第一个商品的信息，请求过来后，第一步会从缓存中取获取，如果缓存中存在的话就会直接返回；如果不存在的话，它会从MySQL也就是数据库中获取，从数据库中获取之后，它会将这一部分的数据写入缓存中，这样下次再获取商品信息的时候就可以直接从缓存中获取了。开发者对数据库中商品的更新或者删除，那么对应的你也需要对缓存中这一部分的信息进行更新或者删除，即始终保证数据库和缓存中对应数据的一致性。

    接下来有一个问题就是当你将某件商品的价格进行了修改（如将皮蛋粥的价格由0.01元修改为1元），可以通过后台或者是数据库的方式来进行修改，然后再次访问http://localhost/sell/buyer/product/list这个商品链接时，由于它是直接从缓存中获取的，因此皮蛋粥的价格依旧还是0.01元。我们需要的时当数据库中的数据修改时，缓存中的数据也需要同步修改，因此可以在商品信息修改的方法上添加@CachePut(cacheNames = "product",key = "123")注解：

    先说明一下@Cacheable(cacheNames = "product", key = "123")注解和@CachePut(cacheNames = "product", key = "123")注解的区别，@Cacheable注解在第一访问的时候会访问到方法的内容，也就是会返回ResultVO对象，这个时候就会给你设置一个Redis，而Redis中存储的就是返回的内容，即ResultVO对象。而当你下一次访问的时候，它就不会再进入这个方法中了，也就不会执行该方法中的代码。

    @CachePut注解每次访问的时候，还是会执行该方法的逻辑，运行里面的代码，然后将每次返回的对象存储到Redis中。之后运行项目，在后台修改商品的价格，然后点击提交，可以发现系统又抛出了无法序列化的错误，但是问题来了，这个save方法返回的是一个ModelAndView对象，它是系统提供的，无法进行序列化。但是前面也说了@CachePut注解会将每次执行结果的返回内容写入Redis中，但是这里返回的是ModelAndView对象，也就是说返回的对象不对，我们写入Redis中的肯定是ResultVO对象。

    可以这样，使用另外一个注解@CacheEvict(cacheNames = "product", key = "123")来替代它，Evict是驱逐的意思，这里就是删除缓存。它表示当你访问这个方法之后（前提是该方法上面有添加@CacheEvict注解），它会将这个缓存的内容给删除掉。既然删除了，那么这个方法第一次执行的时候就会查询数据库，第二次才会查缓存。

    现在问题来了，我就想每次更新Redis缓存中的数据，而不是每次删除然后重新缓存，那么这就需要@CachePut注解方法返回的对象和@Cacheable注解返回的对象是一致的才行。注释掉之前添加的上述三个注解，接下来就找两个返回统一对象的方法，这里以ProductInfoServiceImpl类中的findById和save方法为例，因为它们返回的是同一个对象ProductInfo。首先ProductInfo对象实现Serializable接口并生成序列化ID，然后在findById方法上添加@Cacheable(cacheNames = "product", key = "123")注解，接着在save方法上添加@CachePut(cacheNames = "product", key = "123")注解。最后运行项目，对其进行测试，可以发现更新后，会直接从缓存中获取，不会从数据库中获取，且从缓存中获取的是更新后的数据信息。
